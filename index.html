What Changed

Hardened NanoKVM keeps the upstream hardware stack and focuses on backend, update, and administration hardening.

Rust-only web backend

Release packages replace the privileged Go server with the Hardened Rust backend and reject legacy Go backend files in shipped artifacts.

Hardened authentication

First-boot account setup, CSRF binding, Origin checks, session revocation, login lockout, Argon2id passwords, and security headers.

Signed updates

Application and system channels use signed metadata, archive verification, guarded installs, and explicit separation between app and raw system updates.

System controls

GUI pages for time/NTP, system logs, remote syslog, firewall modes, and network settings under Settings > System.

Managed firewall modes

Baseline, Restricted, and Paranoid profiles make online update, SSH, HTTPS, DNS, NTP, syslog, and WebRTC access explicit.

Video path stability

H.264 Direct is the preferred low-CPU path when HTTPS and WebCodecs are available; MJPEG remains available as a fallback.

Update Model

Two update paths exist because replacing kvmapp is much safer than writing raw SD-card partitions.

Application updates

Replace the Rust server, web assets, helper scripts, init scripts, keys, version files, and kvmapp-side defaults.

Raw system updates

Write boot and rootfs partition images to the SD card. These updates can change the base system and require guarded recovery assumptions.

Offline recovery

Keep physical SD-card recovery access. Raw partition updates have no automatic rollback if the device cannot boot.

Hardened NanoKVM is beta/lab firmware. Read the release notes before installing, keep the original SD card or a known-good backup, and do not enable raw system updates without physical recovery access.
Hardened NanoKVM is an independent security-focused fork for Sipeed NanoKVM. NanoKVM hardware documentation remains available from Sipeed.
